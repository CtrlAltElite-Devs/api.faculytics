# Downstream score-aggregation audit

This audit enumerates every in-tree usage of `positiveScore` / `neutralScore` /
`negativeScore` (or their `positive_score` / `neutral_score` / `negative_score`
SQL mirrors) and judges whether mixing one-hot vLLM-served rows with continuous
OpenAI-served rows distorts each metric.

## Summary

- **Argmax-label buckets** (all matviews, sentiment distribution counts, `sr.label = 'X'` filters) are **unaffected** by mixing one-hot and continuous scores. Argmax of `{1,0,0}` equals argmax of `{0.85,0.1,0.05}` — both resolve to `positive`.
- **Mean / distribution / variance over raw scores** — **not used anywhere in the current codebase**. If added in future work, this audit will need to be re-run.
- **Strength-ranking for sample-quote selection** (two call sites, `analytics.service.ts` + `recommendation-generation.service.ts`) does show an observable asymmetry where vLLM rows always sort first. **Accepted for v1** as a quality-of-ranking concern, not a correctness issue. The `rawResult.servedBy` JSONB field makes a future fix straightforward if the asymmetry becomes visible in dean UX.

## Recommendation

**No code changes required before enabling vLLM in production.** The one pre-existing rankable metric that IS affected (strength-ranked quote selection) is a judgment call on ranking quality, not metric correctness.

## Follow-up

- If the `servedBy: 'openai'` rate surges post-rollout (indicating stale URL or unhealthy vLLM), the strength-ranking asymmetry disappears automatically — the mix reverts to all-continuous scores.
- If a future dashboard adds `AVG(positiveScore)` or similar, this audit must be re-run and the dashboard must either (a) filter by `rawResult->>'servedBy' = 'openai'` OR (b) accept that the mean will shift toward the endpoints as vLLM share increases.
